# 调试代码工作流

## 工具使用原则

⚠️ **必读**：请先阅读 [工具使用指南](tool_usage_guide.md)，了解工具优先级和最佳实践。

**核心原则：优先使用 repo_map、LSP、AST 等高效工具，避免逐个文件搜索！**

## 任务目标

定位和修复代码中的 Bug，解决程序运行错误或异常行为。

## 执行步骤

### 1. 理解问题

**明确问题的具体表现**：
- 什么操作触发了问题？
- 期望的行为是什么？
- 实际的行为是什么？
- 有错误信息吗？
- 能稳定复现吗？

### 2. 收集信息

**⚠️ 重要：优先使用 repo_map 和 LSP 工具获取全局信息！**

参考：[工具使用指南](tool_usage_guide.md)

#### 2.1 查看错误信息
如果用户提供了错误信息：
- 分析错误类型（语法错误、运行时错误、逻辑错误）
- 定位错误位置（文件名、行号）
- 理解错误原因

#### 2.2 获取项目代码地图（推荐，了解上下文）
```
使用工具：repo_map
参数：
  - repo_path: "."
  - max_depth: 3
  - include_tests: false

作用：
- ✅ 快速了解出错代码在项目中的位置
- ✅ 查看相关模块和依赖关系
- ✅ 了解是否有类似的正确实现可以参考
```

#### 2.3 读取相关代码
```
使用工具：read_file
参数：
  - file_path: "出错的文件"

作用：查看出错的代码
```

#### 2.4 检查语法和类型错误（LSP 优先）
```
使用工具：lsp_diagnostics
参数：
  - file_path: "出错的文件"

作用：
- ✅ 查找所有语法、类型错误
- ✅ 获取详细的错误信息和位置
- ✅ 比手动检查更准确
```

#### 2.5 查找引用关系（LSP 优先）
```
使用工具：lsp_find_references
参数：
  - file_path: "出错的文件"
  - line: 出错函数的行号
  - character: 列号

作用：
- ✅ 了解代码的调用关系
- ✅ 查看哪些地方使用了出错的代码
- ✅ 评估修复的影响范围
```

### 3. 分析问题

**系统化地分析问题**：

#### 3.1 确定问题类型
- **语法错误**：代码写错了（拼写、语法）
- **类型错误**：类型不匹配
- **逻辑错误**：代码逻辑有问题
- **运行时错误**：空指针、数组越界等
- **性能问题**：代码太慢

#### 3.2 定位问题位置
- 错误发生在哪个文件？
- 错误发生在哪个函数？
- 错误发生在哪一行？
- 是什么导致了错误？

#### 3.3 分析根本原因
- 为什么会出现这个错误？
- 是代码逻辑问题还是数据问题？
- 是边界条件没处理还是异常没捕获？

### 4. 制定修复方案

**根据问题类型制定方案**：

#### 4.1 语法/类型错误
- 修正拼写错误
- 修正类型错误
- 添加缺失的导入

#### 4.2 逻辑错误
- 修正条件判断
- 修正循环逻辑
- 修正计算公式

#### 4.3 运行时错误
- 添加空值检查
- 添加边界检查
- 添加异常处理

#### 4.4 性能问题
- 优化算法
- 添加缓存
- 减少重复计算

### 5. 修复代码

**使用合适的工具修复**：

#### 5.1 小改动使用 search_replace
```
使用工具：search_replace
参数：
  - file_path: "目标文件"
  - search: "错误的代码"
  - replace: "修复后的代码"

作用：精确替换错误代码
```

#### 5.2 大改动使用 write_file
```
使用工具：write_file
参数：
  - file_path: "目标文件"
  - content: "修复后的完整代码"

作用：重写文件
```

### 6. 验证修复

**确保问题已解决**：

#### 6.1 检查语法
```
使用工具：lsp_diagnostics
参数：
  - file_path: "修复的文件"

作用：确保没有语法错误
```

#### 6.2 查看变更
```
使用工具：git_diff
参数：
  - file_path: "修复的文件"

作用：确认修改正确
```

#### 6.3 运行测试
```
使用工具：run_test
参数：
  - test_path: "相关测试"

作用：验证功能正常
```

### 7. 说明修复

简要说明问题原因和修复方法，根据用户需求灵活调整说明的详细程度。

## 常见问题类型

### 类型1：语法错误

**特征**：代码无法运行，有明确的错误提示

**处理**：
1. 使用 lsp_diagnostics 查找错误
2. 修正语法错误
3. 验证修复

### 类型2：空指针/未定义

**特征**：NoneType、undefined、null reference

**处理**：
1. 找到空值来源
2. 添加空值检查
3. 或者确保值被正确初始化

### 类型3：数组越界

**特征**：IndexError、ArrayIndexOutOfBounds

**处理**：
1. 检查数组访问逻辑
2. 添加边界检查
3. 或者修正索引计算

### 类型4：类型不匹配

**特征**：TypeError、类型转换错误

**处理**：
1. 检查变量类型
2. 添加类型转换
3. 或者修正类型声明

### 类型5：逻辑错误

**特征**：程序运行但结果不对

**处理**：
1. 分析代码逻辑
2. 找到逻辑错误
3. 修正条件或计算

### 类型6：异常未捕获

**特征**：程序崩溃，有异常堆栈

**处理**：
1. 找到抛出异常的位置
2. 添加 try-catch
3. 或者修正导致异常的代码

## 调试技巧

### 技巧1：二分查找

如果不确定问题位置：
1. 注释掉一半代码
2. 看问题是否还存在
3. 缩小范围，重复步骤

### 技巧2：添加日志

在关键位置添加日志：
- 函数入口：记录参数
- 关键步骤：记录中间结果
- 函数出口：记录返回值

### 技巧3：检查边界条件

常见的边界条件：
- 空值（null、None、undefined）
- 空集合（空数组、空字符串）
- 零值（0、0.0）
- 负数
- 最大值/最小值

### 技巧4：检查假设

检查代码中的假设：
- 参数一定有值吗？
- 数组一定不为空吗？
- 文件一定存在吗？
- 网络一定通吗？

## 注意事项

### 修复原则

1. **理解后再修复**：不要盲目修改
2. **最小化修改**：只修复问题，不做额外改动
3. **保持一致性**：修复要符合代码风格
4. **添加防御代码**：避免类似问题再次发生

### 避免的错误

❌ **不要盲目修改**
- 必须理解问题根源
- 不要只是让错误消失

❌ **不要过度修复**
- 只修复当前问题
- 不要顺便重构代码

❌ **不要忽略测试**
- 修复后必须验证
- 确保没有引入新问题

❌ **不要隐藏错误**
- 不要只是捕获异常然后忽略
- 要正确处理错误

### 何时寻求帮助

- 问题太复杂，超出能力范围
- 涉及不熟悉的技术栈
- 需要深入的系统知识
- 可能是环境或配置问题

## 成功标准

- ✅ 问题已解决
- ✅ 代码可以正常运行
- ✅ 测试通过
- ✅ 没有引入新问题
- ✅ 用户理解了问题原因
- ✅ 添加了防御代码避免再次发生
