# 编写代码工作流

## 工具使用原则

⚠️ **必读**：请先阅读 [工具使用指南](tool_usage_guide.md)，了解工具优先级和最佳实践。

**核心原则：优先使用 repo_map、LSP、AST 等高效工具，避免逐个文件搜索！**

## 任务目标

根据用户需求，编写新的代码或实现新功能。

## 执行步骤

### 1. 理解需求

仔细分析用户的需求：
- 要实现什么功能？
- 有哪些具体要求？
- 需要考虑哪些边界情况？
- 是否有性能、安全等特殊要求？

### 2. 了解现有代码

**⚠️ 重要：优先使用 repo_map 获取全局视图，不要逐个文件搜索！**

参考：[工具使用指南](tool_usage_guide.md)

#### 2.1 获取项目代码地图（必须，第一步）
```
使用工具：repo_map
参数：
  - repo_path: "."
  - max_depth: 3
  - include_tests: false

作用：
- ✅ 一次性获取整个项目的代码结构
- ✅ 查看所有类、函数、模块的定义和引用关系
- ✅ 快速了解是否已有类似实现
- ✅ 了解代码组织方式和模块划分
- ✅ 比逐个文件搜索高效 100 倍

这是最重要的一步，必须首先执行！
```

#### 2.2 查找相关代码（可选，repo_map 不够时）
```
使用工具：text_search 或 semantic_code_search
参数：
  - query: "具体的类名或函数名"（不要搜索 "def" 或 "class"）
  - file_pattern: "**/*.py"（根据语言调整）

作用：查找具体的代码位置

注意：repo_map 已经包含了所有定义，通常不需要再搜索
```

#### 2.3 查看相关文件（可选，需要了解细节时）
```
使用工具：read_file 或 batch_read_files
参数：
  - file_path: "从 repo_map 或搜索结果中获取的文件路径"

作用：了解现有代码风格、设计模式

注意：不要推测文件路径，必须从 repo_map 或搜索结果中获取
```

### 3. 设计实现方案

根据现有代码和需求，设计实现方案：

#### 3.1 确定文件位置
- 新建文件还是修改现有文件？
- 文件应该放在哪个目录？
- 文件命名是否符合项目规范？

#### 3.2 确定代码结构
- 需要哪些类/函数？
- 如何组织代码？
- 如何与现有代码集成？

#### 3.3 考虑依赖
- 需要导入哪些模块？
- 是否需要新的依赖？
- 如何处理循环依赖？

### 4. 编写代码

**按照项目现有的代码风格编写**：

#### 4.1 遵循项目规范
- 缩进风格（空格/Tab）
- 命名规范（驼峰/下划线）
- 注释风格
- 文档字符串格式

#### 4.2 编写清晰的代码
- 函数职责单一
- 变量命名清晰
- 适当的注释
- 合理的错误处理

#### 4.3 使用工具写入代码
```
单个文件：
使用工具：write_file
参数：
  - file_path: "目标文件路径"
  - content: "完整的文件内容"

多个文件：
使用工具：batch_write_files
参数：
  - files: [
      {path: "文件1", content: "内容1"},
      {path: "文件2", content: "内容2"}
    ]
```

### 5. 验证代码

**编写代码后，必须验证**：

#### 5.1 检查语法错误
```
使用工具：lsp_diagnostics
参数：
  - file_path: "刚写的文件路径"

作用：检查语法错误、类型错误
```

#### 5.2 查看 Git 变更
```
使用工具：git_diff
参数：
  - file_path: "刚写的文件路径"（可选）

作用：确认修改内容正确
```

#### 5.3 说明实现
简要说明实现的功能和使用方法，根据用户需求灵活调整说明的详细程度。

## 工具使用原则

### 编写代码前

1. **必须先搜索**：避免重复实现
2. **必须先了解结构**：确保代码放在正确位置
3. **必须先看现有代码**：保持代码风格一致

### 编写代码时

1. **一次性写完整**：不要分多次写入同一个文件
2. **包含必要的导入**：确保代码可以运行
3. **添加适当的注释**：帮助理解代码

### 编写代码后

1. **必须验证语法**：使用 lsp_diagnostics
2. **必须查看变更**：使用 git_diff
3. **必须说明用法**：告诉用户如何使用

## 常见场景

### 场景1：新建文件实现功能

```
步骤：
1. text_search(query="类似功能") - 查找是否已有实现
2. get_repo_structure(repo_path=".") - 确定文件位置
3. read_file(file_path="参考文件") - 了解代码风格
4. write_file(file_path="新文件", content="代码") - 编写代码
5. lsp_diagnostics(file_path="新文件") - 验证语法
6. git_diff() - 查看变更
7. 向用户说明实现
```

### 场景2：修改现有文件添加功能

```
步骤：
1. read_file(file_path="目标文件") - 读取现有代码
2. get_file_symbols(file_path="目标文件") - 了解文件结构
3. write_file(file_path="目标文件", content="修改后的代码") - 写入修改
4. lsp_diagnostics(file_path="目标文件") - 验证语法
5. git_diff(file_path="目标文件") - 查看变更
6. 向用户说明修改
```

### 场景3：多文件实现功能

```
步骤：
1. 分析需求，确定需要哪些文件
2. 对每个文件：
   - 查找参考代码
   - 了解代码风格
3. batch_write_files(files=[...]) - 批量写入
4. 对每个文件验证语法
5. git_diff() - 查看所有变更
6. 向用户说明实现
```

## 注意事项

### 代码质量

1. **可读性优先**：清晰的代码比聪明的代码更重要
2. **遵循规范**：保持与项目现有代码风格一致
3. **适当注释**：复杂逻辑需要注释说明
4. **错误处理**：考虑异常情况和边界条件

### 安全考虑

1. **输入验证**：验证用户输入
2. **权限检查**：检查操作权限
3. **敏感信息**：不要硬编码密码、密钥
4. **SQL 注入**：使用参数化查询

### 性能考虑

1. **避免重复计算**：缓存计算结果
2. **避免 N+1 查询**：批量查询数据
3. **合理的数据结构**：选择合适的数据结构
4. **异步处理**：耗时操作考虑异步

### 常见错误

❌ **不要直接写代码**
- 必须先了解现有代码结构
- 必须先查找是否已有类似实现

❌ **不要忽略代码风格**
- 必须保持与项目一致的风格
- 必须遵循项目的命名规范

❌ **不要忘记验证**
- 必须使用 lsp_diagnostics 检查语法
- 必须使用 git_diff 查看变更

❌ **不要写不完整的代码**
- 必须包含必要的导入
- 必须包含错误处理
- 必须包含文档字符串

## 成功标准

- ✅ 代码实现了用户需求
- ✅ 代码风格与项目一致
- ✅ 代码没有语法错误
- ✅ 代码有适当的注释
- ✅ 用户知道如何使用
- ✅ 考虑了错误处理和边界情况
