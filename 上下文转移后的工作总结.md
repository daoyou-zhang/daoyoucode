## 核心问题

工作流中的"模式3：拆分大类"部分存在硬编码路径示例，导致：
1. 无法应用于不同项目
2. 在不同目录运行时路径错误
3. LLM 容易直接使用示例中的硬编码路径
4. 缺少动态路径获取和传递的指导

## 解决方案

### 1. 添加"步骤0：定位目标文件"

在所有步骤之前添加了必须执行的步骤0，明确要求先搜索定位文件：

```markdown
#### 步骤0：定位目标文件（必须第一步）⭐⭐⭐

工具序列：
1. 搜索目标文件
   text_search(query="用户提到的类名", file_pattern="**/*.py")
   
2. 提取路径信息
   → target_file = "搜索结果中的路径"
   → target_dir = "文件所在目录"
   
3. 验证文件存在
   read_file(file_path=target_file)
```

### 2. 添加"路径处理规则"部分

在"模式3"开头添加了详细的路径处理规则：

```markdown
## 路径处理规则 ⚠️⚠️⚠️

### 核心原则：永远不要硬编码路径

❌ 错误做法：
write_file(file_path="backend/daoyoucode/agents/core/config.py", ...)

✅ 正确做法：
1. 使用 text_search 定位文件
2. 从搜索结果提取路径
3. 使用 read_file 验证路径
4. 基于已知路径构造新路径
5. 所有后续步骤使用提取的路径
```

### 3. 重写所有步骤使用路径变量

将所有步骤改为使用从步骤0提取的路径变量：

**步骤1：分析类的职责**
- 使用 `target_file` 变量
- 明确标注"⚠️ 使用步骤0获取的 target_file"

**步骤2：创建配置文件**
- 基于 `target_dir` 构造 `config_file`
- 示例：`config_file = target_dir + "/config.py"`
- 使用构造的路径变量，不硬编码

**步骤3：创建第一个新类**
- 基于 `target_dir` 构造 `llm_caller_file`
- 使用构造的路径变量

**步骤4：更新原类使用新类**
- 使用 `target_file` 变量
- 所有 search_replace 调用都使用路径变量

**步骤5：查找并更新引用**
- 使用 `target_file` 查找引用
- 从 lsp_find_references 结果中提取引用路径
- 使用提取的引用路径变量（ref_file_1, ref_file_2, ...）

## 路径传递模式

### 完整流程

```
阶段1：定位
  text_search
      ↓
  提取路径
      ↓
  target_file, target_dir
      ↓
  read_file（验证）

阶段2：创建
  构造新路径
      ↓
  config_file = target_dir + "/config.py"
      ↓
  write_file
      ↓
  lsp_diagnostics（验证）

阶段3：更新
  search_replace(file_path=target_file, ...)
      ↓
  lsp_diagnostics（验证）

阶段4：引用
  lsp_find_references(file_path=target_file, ...)
      ↓
  提取引用路径
      ↓
  ref_file_1, ref_file_2, ...
      ↓
  search_replace(file_path=ref_file_1, ...)
      ↓
  lsp_diagnostics（验证）
```

### 关键点

1. **搜索是起点** - 所有路径都从搜索开始
2. **提取是关键** - 从搜索结果中提取路径
3. **验证是保障** - 每次使用前都验证
4. **构造是规则** - 基于已知路径构造新路径
5. **变量是载体** - 使用变量传递路径
