# 工具参考手册

本文档列出所有可用工具及其功能，供编写工作流时参考。

---

## 📁 文件操作工具（9个）

### read_file
**功能**：读取单个文件内容

**用途**：
- 查看文件内容
- 理解代码实现
- 获取配置信息

**使用场景**：
- 需要查看具体文件内容时
- 已知文件路径时

**注意事项**：
- 先搜索定位文件，再读取
- 避免盲目读取大量文件

---

### write_file
**功能**：写入单个文件

**用途**：
- 创建新文件
- 修改现有文件
- 保存代码

**使用场景**：
- 编写新代码
- 修改配置文件
- 创建文档

**注意事项**：
- 写入前先读取原文件（如果存在）
- 确保路径正确
- 写入后验证（使用 lsp_diagnostics 或 run_lint）

---

### batch_read_files
**功能**：批量读取多个文件

**用途**：
- 同时读取多个相关文件
- 提高效率

**使用场景**：
- 需要查看多个文件时
- 分析模块间关系时

**注意事项**：
- 不要一次读取太多文件
- 先确定需要读取的文件列表

---

### batch_write_files
**功能**：批量写入多个文件

**用途**：
- 同时修改多个文件

**使用场景**：

- 批量生成代码

**注意事项**：
- 确保所有文件路径正确
- 写入后批量验证

---

**用途**：

---

### create_directory

## 🌳 代码结构工具（3个）

### repo_map
**功能**：生成代码地图（核心工具）

**参数**：
- `enable_lsp=true` - 启用LSP，显示引用计数
- `max_tokens` - 控制输出大小

**用途**：
- 了解项目整体结构
- 查看代码引用关系
- 识别核心模块

**使用场景**：
- 理解项目时（必用）
- 查找未使用代码时
- 分析架构时

**优先级**：最高（先全局后细节）

**注意事项**：
- 大项目可能需要调整 max_tokens
- enable_lsp=true 可以看到引用计数（"# X次引用"）
- 引用计数说明：
  - "# 0次引用" = 未使用
  - "# 1次引用" + 私有方法 = 可能未使用
  - "# 2次引用"+ = 正在使用

---

## 🎯 工具使用优先级

### 理解项目
1. discover_project_docs（最高）
2. get_repo_structure（高）
3. repo_map (enable_lsp=true)（高）

### 查找代码
1. semantic_code_search（推荐）
2. text_search / regex_search
3. repo_map

### 读取代码
1. 先搜索定位
2. 再 read_file / batch_read_files

### 编写代码
1. 先了解现有代码（repo_map）
2. write_file / batch_write_files
3. 验证（lsp_diagnostics / run_lint）

### 查找未使用代码
1. repo_map (enable_lsp=true)（查看引用计数）
2. 不需要逐个使用 lsp_find_references

---

## 💡 使用建议

### 原则1：先全局后细节
- 先用 repo_map 了解全局
- 再用 read_file 查看细节

### 原则2：先搜索后读取
- 先用 semantic_code_search 或 text_search 定位
- 再用 read_file 读取

### 原则3：写代码后必须验证
- 使用 lsp_diagnostics 检查错误
- 使用 run_lint 检查规范
- 使用 run_test 验证功能

### 原则4：避免重复工作
- repo_map (enable_lsp=true) 已包含引用计数
- 不需要再用 lsp_find_references 逐个查找

### 原则5：最小化读取
- 只读取必要的文件
- 使用 get_file_symbols 预览文件结构
- 避免读取大量无关文件

## 📋 常见任务工具组合

### 理解项目
```
1. discover_project_docs
2. get_repo_structure
3. repo_map (enable_lsp=true)
```

### 编写代码
```
1. repo_map - 了解现有结构
2. semantic_code_search - 查找相关代码
3. read_file - 读取相关文件
4. write_file - 编写代码
5. lsp_diagnostics - 验证代码
```

### 查找未使用代码
```
1. repo_map (enable_lsp=true) - 查看引用计数
2. 分析 "# 0次引用" 的代码
```

### 重构代码
```
1. repo_map - 了解影响范围
2. read_file - 读取要重构的代码
3. search_replace / lsp_rename - 修改代码
4. lsp_diagnostics - 验证修改
5. run_test - 测试功能
```

### 调试问题
```
1. semantic_code_search - 定位问题代码
2. read_file - 查看代码
3. lsp_diagnostics - 检查错误
4. write_file - 修复代码
5. run_test - 验证修复
```

## ⚠️ 注意事项

### 不要过度使用工具
- 不是每个任务都需要所有工具
- 根据具体情况选择合适的工具

### 不要盲目读取文件
- 先搜索定位，再读取
- 避免读取大量无关文件

### 不要忘记验证
- 写代码后必须验证
- 使用 lsp_diagnostics 或 run_lint

### 不要重复工作
- repo_map 已包含很多信息
- 不需要重复调用其他工具

## 📝 编写工作流时的参考

在编写工作流时，参考本文档：
1. 选择合适的工具
2. 遵循工具优先级
3. 提供工具使用建议
4. 不要强制执行顺序

**示例**：
```markdown
## 推荐工具
- `repo_map` (enable_lsp=true) - 了解项目结构
- `semantic_code_search` - 搜索相关代码
- `read_file` - 读取文件内容

## 关键原则
- 先全局后细节
- 先搜索后读取
- 写代码后必须验证
```

这样既提供了指导，又不限制LLM的灵活性。
