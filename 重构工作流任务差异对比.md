# 重构工作流任务差异对比

## 核心问题

当前 refactor_code.md 包含多种重构场景，但它们的：
- **任务目标不同**
- **复杂度不同**
- **工具使用不同**
- **步骤流程不同**

混在一起会导致 LLM 混淆，执行效率低。

## 工作流对比矩阵

| 工作流 | 任务目标 | 复杂度 | 主要工具 | 文件操作 | 影响范围 | 典型场景 |
|--------|---------|--------|---------|---------|---------|---------|
| **split_large_class** | 拆分大类为多个小类 | ⭐⭐⭐⭐⭐ | write_file + search_replace + lsp_find_references | 创建多个新文件 + 修改原文件 + 更新引用 | 多文件 | agent.py 1500行 → 拆分为 5 个文件 |
| **extract_class** | 提取功能为新类 | ⭐⭐⭐⭐ | write_file + search_replace + lsp_find_references | 创建1个新文件 + 修改原文件 + 更新引用 | 2-5个文件 | 工具函数 → ToolHelper 类 |
| **extract_function** | 提取代码块为函数 | ⭐⭐ | search_replace | 只修改原文件 | 单文件 | 100行函数 → 拆分为 5 个小函数 |
| **rename_code** | 重命名类/函数/变量 | ⭐⭐⭐ | lsp_find_references + search_replace | 修改多个文件 | 多文件 | BaseAgent → Agent |
| **remove_duplication** | 消除重复代码 | ⭐⭐⭐ | text_search + search_replace | 修改多个文件 | 多文件 | 3处相同代码 → 提取为公共函数 |
| **simplify_conditions** | 简化复杂条件 | ⭐⭐ | search_replace | 只修改原文件 | 单文件 | 5层嵌套 if → 提前返回 |

## 详细任务差异分析

### 1. split_large_class（拆分大类）⭐⭐⭐⭐⭐

#### 任务特征
- **最复杂的重构场景**
- 需要创建多个新文件
- 需要处理复杂的依赖关系
- 需要更新所有引用

#### 典型任务
```
输入：agent.py（1500行，包含 LLM调用、工具管理、记忆管理）

输出：
- agent.py（500行，核心逻辑）
- config.py（配置类）
- llm_caller.py（LLM调用）
- tool_manager.py（工具管理）
- memory_manager.py（记忆管理）
```

#### 关键步骤
1. 定位目标文件（搜索）
2. 分析职责（AST + 手动）
3. 创建配置文件（避免循环导入）
4. 创建第一个新类
5. 更新原类使用新类
6. 查找并更新所有引用
7. 重复 4-6（拆分下一个职责）

#### 工具使用
- text_search - 定位文件
- read_file - 读取和验证
- get_file_symbols - 分析结构
- write_file - 创建新文件（多次）
- search_replace - 修改原文件（多次）
- lsp_find_references - 查找引用
- lsp_diagnostics - 验证（每步）

#### 难点
- 需要处理循环导入
- 需要一次只拆一个职责
- 需要保持代码可运行
- 需要更新所有引用

---

### 2. extract_class（提取类）⭐⭐⭐⭐

#### 任务特征
- 中等复杂度
- 创建一个新文件
- 移动相关功能到新类
- 更新调用方

#### 典型任务
```
输入：utils.py（包含 10 个工具函数）

输出：
- utils.py（保留通用函数）
- file_helper.py（新类 FileHelper，包含文件相关函数）
```

#### 关键步骤
1. 定位目标文件
2. 识别相关函数和数据
3. 设计类的接口
4. 创建新类文件
5. 移动函数到类中
6. 更新原文件的调用
7. 查找并更新其他文件的调用

#### 工具使用
- text_search - 定位文件
- read_file - 读取内容
- get_file_symbols - 分析函数
- write_file - 创建新类（1次）
- search_replace - 修改原文件和调用方
- lsp_find_references - 查找所有调用

#### 难点
- 需要设计合理的类接口
- 需要更新所有调用方
- 需要处理导入关系

#### 与 split_large_class 的区别
| 维度 | split_large_class | extract_class |
|------|-------------------|---------------|
| 起点 | 一个大类 | 一组函数 |
| 终点 | 多个小类 | 一个新类 |
| 新文件数 | 3-5个 | 1个 |
| 复杂度 | 非常高 | 中等 |
| 循环导入风险 | 高（需要先创建config） | 低 |

---

### 3. extract_function（提取函数）⭐⭐

#### 任务特征
- 简单重构
- 不创建新文件
- 只在单个文件内操作
- 快速执行

#### 典型任务
```
输入：
def process_data(data):
    # 100 行代码
    # 包含：验证、转换、计算、保存

输出：
def process_data(data):
    validate_data(data)
    transformed = transform_data(data)
    result = calculate_result(transformed)
    save_result(result)
    
def validate_data(data): ...
def transform_data(data): ...
def calculate_result(data): ...
def save_result(result): ...
```

#### 关键步骤
1. 定位目标文件
2. 识别可提取的代码块
3. 确定函数名和参数
4. 提取为新函数（在同一文件）
5. 替换原代码为函数调用
6. 验证功能不变

#### 工具使用
- text_search - 定位文件
- read_file - 读取内容
- search_replace - 提取函数（多次）
- lsp_diagnostics - 验证

#### 难点
- 需要正确识别参数和返回值
- 需要保持逻辑不变

#### 与其他工作流的区别
| 维度 | extract_function | extract_class | split_large_class |
|------|------------------|---------------|-------------------|
| 新文件 | 0 | 1 | 3-5 |
| 操作范围 | 单文件内 | 跨文件 | 跨多文件 |
| 复杂度 | 低 | 中 | 高 |
| 执行时间 | 2-5分钟 | 10-15分钟 | 30-60分钟 |

---

### 4. rename_code（重命名）⭐⭐⭐

#### 任务特征
- 中等复杂度
- 不创建新文件
- 需要更新所有引用
- 影响范围可能很大

#### 典型任务
```
输入：
class BaseAgent:  # 在 10 个文件中被引用
    ...

输出：
class Agent:  # 所有引用都已更新
    ...
```

#### 关键步骤
1. 定位目标文件
2. 确定新名称
3. 查找所有引用
4. 在每个文件中替换
5. 验证所有文件

#### 工具使用
- text_search - 定位文件
- lsp_find_references - 查找所有引用（关键）
- search_replace - 在每个文件中替换
- lsp_diagnostics - 验证每个文件

#### 难点
- 需要找到所有引用（不能遗漏）
- 需要避免误替换（如字符串中的名称）
- 需要更新导入语句

#### 与其他工作流的区别
| 维度 | rename_code | extract_function | split_large_class |
|------|-------------|------------------|-------------------|
| 代码结构变化 | 无 | 有 | 大 |
| 新增代码 | 无 | 有 | 大量 |
| 主要工作 | 查找替换 | 提取重组 | 拆分重组 |
| 风险点 | 遗漏引用 | 逻辑错误 | 循环导入 |

---

### 5. remove_duplication（消除重复）⭐⭐⭐

#### 任务特征
- 中等复杂度
- 可能创建新文件（如果提取到新模块）
- 需要查找所有重复
- 需要更新多处调用

#### 典型任务
```
输入：
# file1.py
def process_user(user):
    if not user.email:
        raise ValueError("Email required")
    if not user.name:
        raise ValueError("Name required")
    ...

# file2.py
def process_admin(admin):
    if not admin.email:
        raise ValueError("Email required")
    if not admin.name:
        raise ValueError("Name required")
    ...

输出：
# validators.py（新文件）
def validate_user_data(user):
    if not user.email:
        raise ValueError("Email required")
    if not user.name:
        raise ValueError("Name required")

# file1.py
def process_user(user):
    validate_user_data(user)
    ...

# file2.py
def process_admin(admin):
    validate_user_data(admin)
    ...
```

#### 关键步骤
1. 查找重复代码（text_search）
2. 分析重复模式
3. 设计公共函数/类
4. 创建公共模块（如果需要）
5. 提取公共代码
6. 替换所有重复处
7. 验证所有文件

#### 工具使用
- text_search - 查找重复（关键）
- read_file - 读取和比较
- write_file - 创建公共模块（可选）
- search_replace - 替换重复代码
- lsp_diagnostics - 验证

#### 难点
- 需要识别真正的重复（不是表面相似）
- 需要设计合理的抽象
- 需要处理细微差异

#### 与其他工作流的区别
| 维度 | remove_duplication | extract_function | extract_class |
|------|-------------------|------------------|---------------|
| 起点 | 多处重复代码 | 一个长函数 | 一组相关函数 |
| 查找方式 | text_search | 手动识别 | 手动识别 |
| 影响文件数 | 多个 | 1个 | 2-3个 |
| 主要挑战 | 识别重复 | 提取逻辑 | 设计接口 |

---

### 6. simplify_conditions（简化条件）⭐⭐

#### 任务特征
- 简单重构
- 不创建新文件
- 只在单个文件内操作
- 改善可读性

#### 典型任务
```
输入：
def check_permission(user, resource):
    if user.is_active:
        if user.role == "admin":
            return True
        else:
            if resource.owner == user:
                return True
            else:
                if resource.is_public:
                    return True
                else:
                    return False
    else:
        return False

输出：
def check_permission(user, resource):
    if not user.is_active:
        return False
    
    if user.role == "admin":
        return True
    
    if resource.owner == user:
        return True
    
    return resource.is_public
```

#### 关键步骤
1. 定位目标文件
2. 识别复杂条件
3. 应用简化技巧：
   - 提前返回
   - 提取条件为变量
   - 提取条件为函数
   - 使用多态
4. 替换原代码
5. 验证功能不变

#### 工具使用
- text_search - 定位文件
- read_file - 读取内容
- search_replace - 简化条件
- lsp_diagnostics - 验证

#### 难点
- 需要保持逻辑等价
- 需要识别简化机会

#### 与其他工作流的区别
| 维度 | simplify_conditions | extract_function | rename_code |
|------|-------------------|------------------|-------------|
| 代码行数变化 | 减少 | 增加 | 不变 |
| 逻辑变化 | 无（等价） | 无 | 无 |
| 主要目标 | 可读性 | 可维护性 | 清晰性 |
| 风险 | 逻辑错误 | 参数错误 | 遗漏引用 |

## 任务决策树

```
用户需求：重构代码
    ↓
问题是什么？
    ↓
├─ 类太大（>500行）？
│   ├─ 是 → split_large_class.md
│   └─ 否 → 继续
│
├─ 有一组相关函数需要组织？
│   ├─ 是 → extract_class.md
│   └─ 否 → 继续
│
├─ 函数太长（>50行）？
│   ├─ 是 → extract_function.md
│   └─ 否 → 继续
│
├─ 命名不清晰？
│   ├─ 是 → rename_code.md
│   └─ 否 → 继续
│
├─ 有重复代码？
│   ├─ 是 → remove_duplication.md
│   └─ 否 → 继续
│
└─ 条件逻辑复杂（>3层嵌套）？
    ├─ 是 → simplify_conditions.md
    └─ 否 → refactor_code.md（通用）
```

## 适用场景对比

### split_large_class
**适用**：
- ✅ 类超过 500 行
- ✅ 类承担多个职责
- ✅ 类难以理解和维护
- ✅ 有充足时间（30-60分钟）

**不适用**：
- ❌ 类职责单一
- ❌ 类已经很小（<200行）
- ❌ 时间紧迫

### extract_class
**适用**：
- ✅ 有一组相关函数
- ✅ 函数共享数据
- ✅ 需要封装状态

**不适用**：
- ❌ 函数不相关
- ❌ 函数是纯函数（无状态）

### extract_function
**适用**：
- ✅ 函数超过 50 行
- ✅ 函数做多件事
- ✅ 有明显的代码块

**不适用**：
- ❌ 函数已经很小
- ❌ 代码块高度耦合

### rename_code
**适用**：
- ✅ 命名不清晰
- ✅ 命名不符合规范
- ✅ 需要统一命名

**不适用**：
- ❌ 命名已经很好
- ❌ 会造成大量修改

### remove_duplication
**适用**：
- ✅ 有 3 处以上重复
- ✅ 重复代码相似度高
- ✅ 重复代码稳定

**不适用**：
- ❌ 只有 2 处重复
- ❌ 重复代码差异大
- ❌ 重复代码可能分化

### simplify_conditions
**适用**：
- ✅ 条件嵌套超过 3 层
- ✅ 条件逻辑复杂
- ✅ 难以理解

**不适用**：
- ❌ 条件已经简单
- ❌ 条件逻辑清晰

## 总结

### 为什么需要拆分？

1. **任务目标完全不同**
   - split_large_class：拆分 → 创建多个文件
   - extract_function：提取 → 在同一文件内
   - rename_code：重命名 → 不改变结构

2. **复杂度差异巨大**
   - split_large_class：⭐⭐⭐⭐⭐（30-60分钟）
   - extract_function：⭐⭐（5-10分钟）
   - 差异 6-12 倍

3. **工具使用不同**
   - split_large_class：需要 write_file（多次）
   - extract_function：只需要 search_replace
   - rename_code：核心是 lsp_find_references

4. **步骤流程不同**
   - split_large_class：20-40 步，需要循环
   - extract_function：5-8 步，线性流程
   - 混在一起会让 LLM 混淆

5. **风险等级不同**
   - split_large_class：⚠️⚠️⚠️⚠️⚠️（极高风险）
   - extract_function：⚠️⚠️（低风险）
   - 需要不同的验证策略

### 拆分后的优势

1. **LLM 更容易理解**
   - 每个工作流职责单一
   - 步骤清晰，不会混淆

2. **执行成功率更高**
   - 针对性的指导
   - 减少不必要的步骤

3. **维护更容易**
   - 每个文件独立优化
   - 修改影响范围小

4. **用户体验更好**
   - 意图匹配更精确
   - 执行速度更快

### 建议

✅ **强烈建议拆分**，因为这些工作流：
- 任务目标不同
- 复杂度差异大
- 工具使用不同
- 步骤流程不同
- 风险等级不同

混在一起只会降低效率和成功率。
